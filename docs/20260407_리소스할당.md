# Resource Allocation 및 root.temp 큐 테스트 문서

**테스트 일시**: 2026년 4월 7일
**목적**: Resource Allocation 기능을 통해 common 네임스페이스의 root.temp 큐에 SparkApplication을 생성하는지 검증

---

## 1. 개요

### 1.1 테스트 목표

Hynix Service의 Resource Allocation 기능을 통해 다음 조건을 검증:

1. **source 큐 측정**: default 네임스페이스의 root.ias 큐 할당율이 지정된 퍼센트 이상인지 확인
2. **target 측정**: common 네임스페이스 리소스쿼터 할당율이 지정된 퍼센트 이하인지 확인
3. **조건 충족 시**: common 네임스페이스 + root.temp 큐에 SparkApplication 생성
4. **조건 불충족 시**: resource_calculation의 tiers 큐에 SparkApplication 생성

### 1.2 테스트 환경

- **Kubernetes Cluster**: minikube (v1.35.0)
- **YuniKorn Scheduler**: v1.0.0 (default 네임스페이스)
- **Spark Operator**: v2.4.0
- **Hynix Service**: Go 1.23, Port 8080
- **테스트 네임스페이스**: common (ResourceQuota: CPU 20 cores, Memory 40Gi)

## 2. 아키텍처 및 구조 변경

### 2.1 기존 구조 문제점

```json
{
  "resource_allocation": {
    "enabled": true,
    "queue": "root.temp",      // ❌ source 측정과 submit 큐가 혼용
    "namespace": "common"
  }
}
```

**문제**: 하나의 `queue` 필드가 source 측정과 submit 큐에 모두 사용되어 source 큐(root.ias)를 측정하면서 root.temp에 submit할 수 없음

### 2.2 변경된 구조

```json
{
  "resource_allocation": {
    "enabled": true,
    "name": "ias allocation",
    "source": {
      "cpu": 50,
      "memory": 50,
      "queue": "root.ias"      // ✅ 할당량 측정용 큐
    },
    "target": {
      "cpu": 5,
      "memory": 5,
      "queue": "root.temp"     // ✅ submit용 큐
    },
    "namespace": "common"        // ✅ 생성할 네임스페이스
  }
}
```

**CalculateResourceAllocation 함수 수정**:

```go
// Source 큐 리소스 사용량 조회 (YuniKorn REST API)
// allocation.Source.Queue에서 큐 경로 추출 (예: "root.ias")
sourceUsage, err := GetQueueResourceUsageFromYuniKorn(allocation.Source.Queue)

// Target 네임스페이스 리소스쿼터 사용량 조회
targetUsage, err := k8sClient.GetNamespaceResourceQuotaUsage(allocation.Namespace)

// 조건 평가
sourceCPUMet := sourceUsage.CPUPercent >= float64(allocation.Source.CPU)
sourceMemoryMet := sourceUsage.MemoryPercent >= float64(allocation.Source.Memory)
targetCPUMet := targetUsage.CPUPercent <= float64(allocation.Target.CPU)
targetMemoryMet := targetUsage.MemoryPercent <= float64(allocation.Target.Memory)

conditionMet := sourceCPUMet && sourceMemoryMet && targetCPUMet && targetMemoryMet

if conditionMet {
    return &ResourceAllocationResult{
        UseAllocation: true,
        Namespace:     allocation.Namespace,
        Queue:         allocation.Target.Queue,  // ✅ target.queue에 submit
        ...
    }, nil
}
```

```go
// ApplyExecutorCPUToYAML - YAML의 executor CPU 관련 필드 업데이트
func ApplyExecutorCPUToYAML(yamlStr string, cpu int) string {
    // CPU 값 관련 플레이스홀더 계산
    executorCPUCores := fmt.Sprintf("%d", cpu)
    executorCPUMinResource := fmt.Sprintf("%dm", cpu*100)
    executorCPULimit := fmt.Sprintf("%d", cpu)
    executorCPURequest := fmt.Sprintf("%dm", cpu*500)

    // 플레이스홀더 치환
    yamlStr = strings.ReplaceAll(yamlStr, "EXECUTOR_CPU_PLACEHOLDER", executorCPUCores)
    yamlStr = strings.ReplaceAll(yamlStr, "EXECUTOR_CPU_MIN_RESOURCE", executorCPUMinResource)
    yamlStr = strings.ReplaceAll(yamlStr, "EXECUTOR_CPU_LIMIT", executorCPULimit)
    yamlStr = strings.ReplaceAll(yamlStr, "EXECUTOR_CPU_REQUEST", executorCPURequest)

    return yamlStr
}
```

### 7.3 MinIO 파일 업로드 테스트

## 8. Arguments 필드 테스트

### 8.1 기능 개요

**목적**: Spark Application에 실행 arguments를 동적으로 전달

**구현**:
- 요청의 `arguments` 필드를 통해 Spark Application 실행 인자 전달
- 공백으로 구분된 여러 인자 지원 (예: `"111 222 333"`)
- `spec:` 바로 다음에 `arguments:` 섹션 동적 추가
- arguments가 비어있거나 없으면 추가하지 않음

### 8.2 코드 변경 사항

**services/template.go - ApplyArgumentsToYAML 함수 수정**:

**변경 전**: `batchSchedulerOptions:` 앞에 arguments 추가
**변경 후**: `spec:` 바로 다음에 arguments 추가

```go
// spec: 라인 찾아서 그 다음에 삽입
lines := strings.Split(yamlStr, "\n")
for i, line := range lines {
    trimmed := strings.TrimSpace(line)
    if strings.HasPrefix(trimmed, "spec:") {
        // spec: 다음 속성들은 2칸 더 들여쓰기
        baseIndent := strings.Repeat(" ", len(line)-len(strings.TrimLeft(line, " ")))
        indent := baseIndent + "  "

        // arguments 라인 생성
        argsLines := []string{fmt.Sprintf("%sarguments:", indent)}
        for _, arg := range argArray {
            argsLines = append(argsLines, fmt.Sprintf("%s  - \"%s\"", indent, arg))
        }

        // lines에 삽입
        newLines := make([]string, 0, len(lines)+len(argsLines))
        newLines = append(newLines, lines[:i+1]...)
        newLines = append(newLines, argsLines...)
        newLines = append(newLines, lines[i+1:]...)

        return strings.Join(newLines, "\n")
    }
}
```

### 11.3 성공 항목

1. **Resource Allocation 활성화/비활성화**
   - `enabled: true` 시 Resource Allocation 로직 실행
   - `enabled: false` 시 Resource Calculation 로직 실행

2. **소스 큐 기반 할당율 측정**
   - YuniKorn REST API를 통한 큐 리소스 사용량 실시간 조회
   - CPU/Memory 할당율 계산 (used / max * 100)

3. **리소스쿼터 기반 할당율 측정**
   - Kubernetes API를 통한 네임스페이스 ResourceQuota 조회
   - 사용량/하드제한 계산

4. **조건부 리소스 할당**
   - source 조건 (최소 사용량) 충족 시
   - target 조건 (여유 공간) 충족 시
   - 둘 다 충족 시 Resource Allocation 사용

5. **동적 큐 선택**
   - Resource Allocation: target.queue 사용
   - Resource Calculation: tiers[].queue 사용

6. **MinIO 기반 리소스 계산**
   - 파일 크기 기반 자동 큐 선택
   - small (< 10MB), medium (>= 10MB, < 50GB), large (>= 50GB)

7. **티어별 CPU 자동 설정**
   - config.json의 tiers[].cpu 필드로 Executor CPU 자동 설정
   - 4개 YAML 필드 자동 계산:
     - spec.executor.cores (CPU 값)
     - task-groups minResource.cpu (CPU × 100m)
     - executor resources.limits.cpu (CPU 값)
     - executor resources.requests.cpu (CPU × 500m)

8. **Arguments 동적 주입**
   - 요청의 arguments 필드로 Spark Application 실행 인자 전달
   - 공백으로 구분된 여러 인자 지원
   - spec: 바로 다음에 arguments: 섹션 동적 추가
   - arguments가 비어있거나 없으면 추가하지 않음
   - 올바른 YAML 들여쓰기 자동 적용

9. **Build Number 버전 관리**
   - config.json의 major, minor, patch 필드로 버전 관리
   - {major}.{minor}.{patch} 형식으로 버전 생성 (예: 4.0.1)
   - Docker 이미지 태그에 자동 적용: docker.io/library/spark:{version}
   - sparkVersion에 자동 적용
   - build-number 라벨에 자동 적용
   - 모든 컨테이너 이미지에 통일된 버전 적용
